## 단일 책임 원칙 (S)

하나의 함수, 하나의 컴포넌트에서는 **한 가지의 행동**만 한다.

<Image url='/img/solid-data1.png' />

## 개방-폐쇄 원칙 (O)

개방-폐쇄 원칙은 원본 소스 코드를 변경하지 않고 확장할 수 있는 방식으로 구성 요소를 구조화하도록 권고합니다
개방-폐쇄 원칙에 따라 컴포넌트 간의 결합을 줄이고 확장성과 재사용성을 높일 수 있습니다.

간단한 예를 들어보자.
목록에서
이전 버튼, 다음 버튼, 목록 버튼의 UI가 동일하지만 기능은 이전

```tsx
<ReadMoreButton onClick={nextFetch}>더 불러오기</ReadMoreButton>
```

## 리스코프 치환 원칙 (L)

상속느낌인데 리액트 팀에서 매우 권장하지 않음... 자체가 컴포넌트 개념인데 상위-하위 관계를 만들어주는 게 별로여서 일까

## 인터페이스 분리 원칙 (I)

인터페이스는 사용하지 않는 데이터에 의존하면 안된다.
프롭스로 데이터 전부를 넘기는 것처럼 사용하는 것보다는 밖에서 특정값만 넘겨주고 컴포넌트는 넘겨준 데이터로 렌더링할 수 있도록 해야함

## 의존관계 역전 원칙 (D)

공통된 추상화에 의존해야한다.
이 코드에서 LoginForm 컴포넌트는 api 모듈을 직접 참조하므로 둘 사이는 긴밀히 결합되어 있습니다
API에 크리덴셜을 제출하는 로직은 onSubmit 콜백을 통해 추상화되었으며 이제 이 로직의 구체적인 구현을 제공하는 것은 상위 컴포넌트의 책임입니다.
